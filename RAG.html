The Task ¶

The task for this MC, is both the creation and evaluation of our own RAG (Retrieval Augmented Generation).

In order to do that we will do the following steps. We...:

  • analyze our data with explorative data analysis
  • clean our data (removing data that is useless/harmful for our RAG)
  • apply different chunking techniques for our data
  • use different embedding databanks
  • experiment with a different numbers of layers
  • choose different metrics, according to which we want to judge our RAG
  • visualize our evaluations and discuss our conclusions

In order to make this jupyter notebook easy to follow, and easy to read, I will comment any action we do and try to visualize and explain my thoughtprocess for actions and approaches I take. That way I will learn more about RAG and NPR and the Jupyter Notebook is easier to read

We now export all of these articles into .txt files. In order to do that, we create the class ArticlesExtractor .

ArticlesExtractor works like this: It reads a CSV file that contains article data, processes each entry, and saves the article content as individual .txt files in a given directory. The filenames are based on the article titles, which are cleaned to remove any characters that are not allowed in filenames.

The class has two main parts:

  • ___title_to_filename()__ is a helper method that turns a title into a valid filename by replacing spaces with underscores and removing characters like slashes, colons, or asterisks.
  • extract_articles_from_csv() is the core method. It opens the CSV file, loops through each row, extracts the article content, and writes it to a text file. If the output folder doesn’t exist, it will be created automatically.

import csv
import os
import re

class ArticlesExtractor:
    """
    ArticlesExtractor is a utility class designed to extract articles from a CSV file and save them as text files.
    Each article's title is sanitized to remove characters not allowed in filenames and spaces are replaced with
    underscores. The articles are then saved to the specified output directory, with the filenames derived from
    their titles.
    """
    def _title_to_filename(self, title: str) -> str:
        """
        Converts an article title into a safe filename by removing characters that are not allowed in filenames
        and replacing spaces with underscores. This method is used internally to sanitize article titles before
        saving them as text files.

        Args:
            title (str): The article's title that needs to be converted into a filename.

        Returns:
            str: A sanitized version of the title that can safely be used as a filename.
        """
        filename = re.sub(r'[<>:"/\\|?*]', '', title.replace(' ', '_'))
        return filename

    def extract_articles_from_csv(self, csv_file_path: str, output_directory: str) -> None:
        """
        Extracts articles from a CSV file and saves them as text files in the specified output directory. The CSV
        file should have at least two columns named "Text" and "Title". Each article is saved as a separate text file,
        with the file name derived from its title. The output directory is created if it does not already exist.

        Args:
            csv_file_path (str): The path to the CSV file containing the articles to be extracted. The CSV file
                                 should be encoded in UTF-8 and contain columns for "Text" and "Title".
            output_directory (str): The path to the directory where the extracted articles will be saved as text
                                    files. This directory will be created if it does not exist.

        Returns:
            None
        """
        os.makedirs(output_directory, exist_ok=True)  # Create the directory if it doesn't exist
        with open(csv_file_path, mode='r', encoding='utf-8') as csv_file:
            csv_reader = csv.DictReader(csv_file)
            for row in csv_reader:
                title = self._title_to_filename(row['title'])  # Sanitize article's title
                content_list = eval(row['content'])  # Convert string representation of list back to a list
                output_file_path = os.path.join(output_directory, f"{title}.txt")
                with open(output_file_path, mode='w', encoding='utf-8') as text_file:
                    for line in content_list:
                        cleaned_line = line.strip('"')  # Remove leading and trailing quotes
                        text_file.write(cleaned_line + "\n")

The databases ¶

In order to build the RAG, we need a DAtabase to store our embeddings. I have chosen 2 different databases here to store our embeddings created by the different chunking strategies and different #layers choice.

The reason for choosing each database are the following:

FAISS ¶

FAISS is a library developed by Facebook AI Research that specializes in efficient similarity search. I chose FAISS because it offers high performance for vector search, especially when working with large sets of embeddings. It allows for both exact and approximate nearest neighbor search, making it well-suited for experiments where retrieval speed and search accuracy are key variables. Since it doesn’t include additional document management layers, it keeps the focus strictly on the quality of the embeddings and the effectiveness of the chunking and layer configurations.

ChromaDB ¶

ChromaDB is a vector database designed specifically for use with large language models and retrieval-augmented generation pipelines. I chose ChromaDB for its simplicity and built-in support for storing both embeddings and associated metadata. It allows for easy experimentation with different document structures, and provides a persistent, lightweight solution that integrates smoothly into Python-based workflows. This makes it a strong choice when evaluating not just retrieval quality, but also how chunking strategies perform in a more complete, application-oriented setup.

Number of layers choice ¶

To find out if using more layers improves our RAG results, we create embeddings with both a 6-layer and a 12-layer model. This lets us compare how the number of layers affects performance.

Our chunking strategies ¶

We experiment with 3 different chunking strategies to see how the way we split text affects the performance of our RAG setup.

Recursive chunking:
This method splits text based on a hierarchy of separators (like paragraphs, sentences, then words). It tries to keep chunks semantically meaningful while staying within a size limit.

  • Advantage: Keeps context better than hard splits.
  • Disadvantage: Can result in uneven chunk sizes.

Token based chunking:
Here, text is split into chunks of a fixed number of tokens (here 600 tokens per chunk). Advantage: Consistent chunk sizes for embeddings. Disadvantage: May split sentences in unnatural places, affecting coherence.

Character based chunking:
This approach splits text into chunks based on a fixed number of characters. Advantage: Simple and fast to apply. Disadvantage: Doesn’t account for token boundaries or meaning, which can lead to poor context retention.

Defining a chunking class ¶

with different functions for chunking approaches¶

The DocumentChunker class is designed to read .txt files from a given directory, apply three different chunking methods to each document, and collect the resulting chunks for further analysis.

  • The init method sets up the input directory and the maximum chunk size.
  • read_documents() reads all .txt files in the specified directory and stores their contents along with filenames.
  • chunk_by_character() splits a text into fixed-size chunks based on a defined number of characters.
  • chunk_by_token() tokenizes the text using NLTK, then groups the tokens into chunks without exceeding the size limit.
  • chunk_recursively() attempts to split text more naturally by breaking at the last space before the size limit, preserving word boundaries where possible.
  • process_documents() runs all three chunking methods on each document, storing both the chunk contents and the length of each chunk for later analysis.
  • plot_histograms() generates histograms of chunk lengths for each strategy, helping visualize how evenly the text is split under each method.

class DocumentChunker:
    def __init__(self, directory, max_size=600):
        self.directory = directory
        self.max_size = max_size

    # Function to read text files from the directory
    def read_documents(self):
        documents = []
        for filename in os.listdir(self.directory):
            if filename.endswith(".txt"):
                filepath = os.path.join(self.directory, filename)
                with open(filepath, 'r', encoding='utf-8') as file:
                    documents.append((filename, file.read()))
        return documents

    # Chunk by character
    def chunk_by_character(self, text):
        return [text[i:i + self.max_size] for i in range(0, len(text), self.max_size)]

    # Chunk by token
    def chunk_by_token(self, text):
        tokens = word_tokenize(text)
        chunks = []
        current_chunk = []
        current_size = 0

        for token in tokens:
            token_size = len(token) + 1  # +1 for space or punctuation
            if current_size + token_size > self.max_size:
                chunks.append(" ".join(current_chunk))
                current_chunk = [token]
                current_size = token_size
            else:
                current_chunk.append(token)
                current_size += token_size

        if current_chunk:  # Add the last chunk
            chunks.append(" ".join(current_chunk))

        return chunks

    # Chunk recursively
    def chunk_recursively(self, text):
        if len(text) <= self.max_size:
            return [text]

        # Split the text into smaller chunks based on size
        chunks = []
        while len(text) > self.max_size:
            # Find the last space before the max_size limit
            split_index = text.rfind(" ", 0, self.max_size)
            if split_index == -1:  # If no space found, split at max_size
                split_index = self.max_size

            chunks.append(text[:split_index])
            text = text[split_index:].lstrip()  # Remove leading spaces for next chunk

        if text:  # Add remaining text
            chunks.append(text)

        return chunks

    # Process all documents in the directory and chunk them
    def process_documents(self):
        documents = self.read_documents()
        chunk_data = {
            'character_chunks': [],
            'token_chunks': [],
            'recursive_chunks': [],
            'all_character_chunks': [],
            'all_token_chunks': [],
            'all_recursive_chunks': []
        }

        for filename, document in documents:
            character_chunks = self.chunk_by_character(document)
            token_chunks = self.chunk_by_token(document)
            recursive_chunks = self.chunk_recursively(document)

            chunk_data['character_chunks'].extend([len(chunk) for chunk in character_chunks])
            chunk_data['token_chunks'].extend([len(chunk) for chunk in token_chunks])
            chunk_data['recursive_chunks'].extend([len(chunk) for chunk in recursive_chunks])

            chunk_data['all_character_chunks'].extend(character_chunks)
            chunk_data['all_token_chunks'].extend(token_chunks)
            chunk_data['all_recursive_chunks'].extend(recursive_chunks)

        return chunk_data

    # Function to plot histogram
    def plot_histograms(self, chunk_data):
        strategies = ['character_chunks', 'token_chunks', 'recursive_chunks']

        for strategy in strategies:
            plt.figure(figsize=(8, 6))
            plt.hist(chunk_data[strategy], bins=20, color='skyblue', edgecolor='black')
            plt.title(f"Distribution of Chunk Lengths ({strategy.replace('_', ' ').title()})")
            plt.xlabel('Chunk Length (characters)')
            plt.ylabel('Frequency')
            plt.grid(True)
            plt.show()

The FIASS database works like this: ¶

The FiassConnector class handles both the creation and loading of a FAISS vector store. It uses a HuggingFace embedding model to convert the chunked documents into vectors. If the chunked_documents are provided, it can initialize a new FAISS database and save it locally. Later, the same database can be loaded again using the load_db method.

Once the database is loaded, the retrieve method allows us to perform similarity search. Given a query, it returns the top k most relevant text chunks based on semantic similarity.

In [25]:
from langchain_community.embeddings import HuggingFaceEmbeddings
from typing import List
from langchain.vectorstores import FAISS


class FiassConnector():
    def __init__(self, embedding_model_name, db_path, chunked_documents = None) -> None:
        self.embedding_model_name = embedding_model_name
        self.db_path = db_path
        self.chunked_documents = chunked_documents
        self.embeddings = HuggingFaceEmbeddings(model_name=self.embedding_model_name)
        self.db = None

    def init_databse(self):
        self.db = FAISS.from_texts(self.chunked_documents, self.embeddings)
        self.db.save_local(self.db_path)

        print(f"FAISS DB saved at {self.db_path}")
        return self.db

    def load_db(self):
        self.db = FAISS.load_local(self.db_path, embeddings=self.embeddings, allow_dangerous_deserialization=True)
        return self.db

    def retrieve(self, query: str, k: int) -> List[str]:
        docs = self.db.similarity_search(query, k=5)

        rag_output = []
        for doc in docs:
            rag_output.append(doc.page_content)

        return rag_output

ChromaDB Database creation ¶

To create the ChromaDBs, we use the below created ChromaConnector class. It handles everything from initializing a new database to adding the embedded text chunks and saving the result.

Since we’re working with a lot of data, we add the documents in batches to avoid memory issues. The init_database() method checks if the number of documents is larger than a set batch size (here: 41,666), and if so, splits them up before adding them. Each batch is processed and added to a persistent Chroma database stored at the given path.

The load_db() method allows us to load an existing ChromaDB from disk, using the same embedding model that was used to create it. This makes it easy to resume work or run retrieval without rebuilding the entire database.

Finally, the retrieve() method takes a query string and returns the top-k most similar chunks using Chroma’s similarity search.

Each ChromaConnector instance is tied to a specific embedding model, a set of chunked documents, and an output path. This setup makes it easy to test different combinations — like chunking strategy (token, character, recursive) and model depth (6-layer vs. 12-layer).

class ChromaConnector:
    def __init__(self, embedding_model_name, db_path, chunked_documents=None) -> None:
        self.embedding_model_name = embedding_model_name
        self.db_path = db_path  # This will be used for persistent storage
        self.chunked_documents = chunked_documents
        self.embeddings = HuggingFaceEmbeddings(model_name=self.embedding_model_name)
        self.db = None

    def init_database(self, batch_size=41666):
        if len(self.chunked_documents) > batch_size:
            chunked_batches = [self.chunked_documents[i:i + batch_size] for i in range(0, len(self.chunked_documents), batch_size)]
        else:
            chunked_batches = [self.chunked_documents]

        # Initialize Chroma with persistent storage
        self.db = Chroma(embedding_function=self.embeddings, persist_directory=self.db_path)
        for i, batch in enumerate(chunked_batches):
            print(f"Processing batch {i+1} of {len(chunked_batches)}")
            self.db.add_texts(batch)

        # Save the database to disk
        # self.db.persist()
        print(f"ChromaDB saved at {self.db_path}")
        return self.db

    def load_db(self):
        # Load ChromaDB from the specified directory
        self.db = Chroma(embedding_function=self.embeddings, persist_directory=self.db_path)
        print(f"ChromaDB loaded from {self.db_path}")
        return self.db

    def retrieve(self, query: str, k: int) -> List[str]:
        docs = self.db.similarity_search(query, k=k)

        rag_output = []
        for doc in docs:
            rag_output.append(doc.page_content)

  • Answer Relevancy:
    Measures how directly the model's answer addresses the question.
    → High score = the answer is on-topic and useful.
    Important because we want responses that are actually helpful, not just fluent.

  • Contextual Precision:
    Measures how much of the retrieved context was actually useful for generating the answer.
    → High score = less irrelevant context was included.
    Helps us understand if our chunking or retrieval strategy is too noisy.

  • Contextual Recall: Measures how much of the necessary context was included in what was retrieved.
    → High score = most of the useful context was retrieved.
    Relevant when we want to avoid missing key facts or explanations.

  • Contextual Relevancy: Measures how relevant the retrieved context is overall to the question.
    → High score = better alignment between question and retrieval.
    Useful to detect whether our database is surfacing good support material.

  • Faithfulness:
    Checks whether the model’s answer stays true to the retrieved context — i.e., no hallucinations.
    → High score = the answer sticks to what was in the context.
    Critical in RAG to ensure the model doesn't make things up.

Answer Relevancy ¶

The overall scores for answer relevancy are high across all configurations, ranging roughly between 0.9 and 0.98. This indicates that most RAG setups are generally capable of producing answers that are relevant to the question, regardless of chunking method, vector store, or embedding model size.

That said, character-based chunking with 6-layer embeddings stands out as the strongest performer. It achieves the highest answer relevancy scores across both Chroma and FAISS, and it clearly outperforms the 12-layer version of the same chunking strategy. The difference is consistent and notable, suggesting that, for answer relevancy alone, the shallower model may be more effective — at least when paired with character-level chunking.

The other chunking methods (recursive and token) also show solid performance, but with slightly lower scores overall. Contextual Relevancy ¶

Contextual relevancy measures how useful the retrieved context was in relation to the final answer. Unlike precision or recall (which focus on overlap or coverage), this metric reflects whether the model actually used the retrieved information meaningfully.

Overall, the scores are lower compared to other metrics, ranging roughly from 0.35 to 0.52. This is expected — the bar for contextual relevancy is higher, as it reflects not just what's retrieved, but whether it's integrated well into the model’s response.

As with previous metrics, L6 embeddings outperform L12 across the board for this metric.

These results suggest that Chroma tends to retrieve context that is more tightly aligned with how the model forms its answers, even if FAISS may retrieve more complete or precise chunks. This makes contextual relevancy a helpful complement to recall and precision — highlighting not just what’s retrieved, but what ends up being useful.

Chroma with recursive L12 is also excellent — the highest-scoring Chroma config overall.

There are some small variations (e.g., FAISS outperforms Chroma slightly in most character- and token-based settings), but none are drastic — most setups produce highly faithful outputs.

This suggests that once a RAG system retrieves a reasonably relevant context, GPT-4 does a good job at sticking to the facts, regardless of chunking style or embedding depth.

class FiassConnector():
    def __init__(self, db_path) -> None:
        self.db_path = db_path
        self.embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L12-v2")
        self.db = FAISS.load_local(self.db_path, embeddings=self.embeddings, allow_dangerous_deserialization=True)

    def retrieve(self, query: str, k: int):
        docs = self.db.similarity_search(query, k=k)

        rag_output = []
        for doc in docs:
            rag_output.append(doc.page_content)

        return rag_output
